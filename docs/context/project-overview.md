# 과제 B - 크리에이터 정산 API 개요

## 프로젝트 개요
이 프로젝트는 강의를 판매하는 크리에이터를 위한 정산 API를 구현하는 백엔드 과제이다.  
플랫폼에서 발생한 판매 내역과 취소 내역을 관리하고, 이를 기준으로 크리에이터별 월 정산 금액과 운영자용 전체 정산 현황을 계산하는 것이 핵심 목적이다.

## 현재 프로젝트 범위 구분
### 과제 원문 기준 필수 범위
- 판매 내역 관리
- 취소 내역 관리
- 크리에이터 월별 정산 조회
- 운영자용 기간별 정산 집계
- 정산 규칙과 샘플 데이터 기준 검증

### 추가로 반영할 요구
- 크리에이터 로그인
- 운영자 로그인
- 이후 구현에서는 과제 원문 범위와 별도로, 인증 기능도 함께 고려한다.

## 배경 시나리오
- 크리에이터는 강의를 개설하고 수강생에게 판매한다.
- 플랫폼은 판매 금액에서 수수료를 공제한 뒤 크리에이터에게 정산한다.
- 크리에이터는 자신의 월별 정산 내역을 조회할 수 있어야 한다.
- 운영자는 특정 기간 동안 전체 크리에이터의 정산 현황을 집계할 수 있어야 한다.
- 수강 취소가 발생하면 기존 정산 금액에서 환불 금액이 차감되어야 한다.

## 과제에서 드러나는 핵심 대상
### 1. 판매 내역 `SaleRecord`
- 강의 ID
- 수강생 ID
- 결제 금액
- 판매 당시 적용된 수수료율
- 결제 완료 일시
- 크리에이터 식별 정보

### 2. 취소 내역 `CancelRecord` 또는 동등한 모델
- 원본 판매 내역 참조
- 환불 금액
- 취소 일시

## 필수 구현 범위
### 1. 판매 내역 관리
- 판매 내역 등록
- 취소 내역 등록
- 크리에이터별 판매 내역 목록 조회
- 기간 조건을 포함한 판매/취소 조회

### 2. 크리에이터 월별 정산 조회 API
요청 예시:
- `creatorId`
- `yearMonth` 예: `2025-03`

응답에 포함되어야 하는 항목:
- 해당 월 총 판매 금액
- 취소/환불 금액
- 순 판매 금액 = 총 판매 금액 - 환불 금액
- 플랫폼 수수료 = 수수료율별 순액에 대한 수수료 합계
- 정산 예정 금액 = 순 판매 금액 - 플랫폼 수수료
- 판매 건수
- 취소 건수

### 3. 운영자용 정산 집계 API
- 조회 기간: 시작일 ~ 종료일
- 응답: 크리에이터별 정산 예정 금액 목록
- 응답: 전체 합계

## 샘플 데이터 검증 기준
- `creator-1`, `2025-03`
  - 총 판매 `260000`
  - 환불 `110000`
  - 순 판매 `150000`
  - 수수료 `30000`
  - 정산 예정 `120000`
- 부분 환불
  - `sale-4`와 연결된 취소는 `30000`만 차감되어야 한다.
- 월 경계 처리
  - `sale-5`는 2025-01 판매로 반영된다.
  - 연결 취소는 2025-02 환불로 반영된다.
- 빈 월 조회
  - `creator-3`, `2025-03` 조회 시 0원 기준 응답 또는 동일한 정책으로 일관 처리한다.

## 정산 규칙
- 판매 집계 기준은 결제 완료 일시이다.
- 취소 집계 기준은 취소 일시이다.
- 월별 정산 경계는 KST 기준으로 계산한다.
- 월 범위는 해당 월 1일 `00:00:00`부터 말일 `23:59:59`까지로 본다.
- 환불이 발생하면 이미 계산된 판매 금액에서 환불 금액을 차감해 순 판매 금액을 계산한다.
- 판매 등록 시점의 현재 수수료율은 `SaleRecord`에 스냅샷으로 저장한다.
- 판매분 수수료는 해당 판매에 저장된 수수료율로 계산한다.
- 환불분 수수료 차감은 취소 시점의 현재 수수료율이 아니라 원본 판매에 저장된 수수료율로 계산한다.

## 추가 정책 결정
- 기본 수수료율은 과제 원문에 맞춰 `20%`로 시작하되, 이후 변경 가능성을 고려해 현재 수수료율 설정과 변경 이력을 분리한다.
- 현재 수수료율은 `FeePolicy`에 저장하고, 판매 등록 시점의 현재 수수료율을 `SaleRecord`에 저장한다.
- 운영자가 수수료율을 변경하면 변경 시점부터 즉시 적용되며, 그 이후 등록되는 판매는 변경된 수수료율을 사용한다.
- `FeePolicyHistory`는 수수료율 변경 시점과 변경값을 남기는 이력 조회용이며, 정산 계산의 직접 기준으로 사용하지 않는다.
- 모든 판매의 수수료율이 같으면 아래 과제 원문의 공식과 동일한 결과가 된다.
  - 순 판매 금액 = 총 판매 금액 - 환불 금액
  - 플랫폼 수수료 = 순 판매 금액의 수수료율%
- 플랫폼 수수료는 음수가 될 수 없으므로 최소 0원으로 처리한다.
- 정산 예정 금액이 음수인 경우, 이는 해당 정산에서 크리에이터에게 지급할 금액이 없고 환불 등으로 인해 정산 기준의 차감 금액이 발생했음을 의미한다.
- 이 프로젝트에서는 음수 정산 예정 금액을 이후 정산으로 이월하지 않고, 해당 정산 결과에 직접 반영되는 정산 조정 금액으로 본다.

## 구현 시 주의사항
- 시간대는 반드시 KST 기준으로 일관되게 처리한다.
- API 입력 일시가 `OffsetDateTime`이어도 저장과 정산 계산에 사용하는 내부 값은 KST(`Asia/Seoul`) 기준 `LocalDateTime`으로 정규화해 처리한다.
- 정산 계산 로직은 판매와 취소의 기준 시점이 다르다는 점을 분리해서 구현한다.
- 정산 수수료 계산은 전체 순 판매 금액에 단일 수수료율을 곱하는 방식이 아니라, 판매/환불의 원본 판매 수수료율을 기준으로 수수료율별 순액을 만든 뒤 합산한다.
- 판매는 `paidAt`, 취소는 `canceledAt` 기준으로 해당 월에 반영하고, 수수료율은 판매 당시 `SaleRecord.feeRatePercent`를 기준으로 적용한다.
- 운영자 집계 API와 크리에이터 월별 조회 API는 동일한 정산 규칙을 공유해야 한다.
- 금액 계산은 정밀도가 중요하므로 정수 금액 또는 적절한 금액 타입 사용을 우선 검토한다.

## 이 문서의 목적
이 문서는 이 프로젝트의 전체 범위를 설명하기 위한 문서이며, 과제 원문 기준 필수 요구와 추가로 반영할 요구를 함께 구분해서 관리하기 위한 기준 문서이다.
